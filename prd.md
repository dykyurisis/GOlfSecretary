## 1. 배경 / 문제 정의

부부 2명이 미국 오렌지카운티의 회원제 골프장 3곳(**Coto de Caza, Aliso Viejo, Old Ranch**, 모두 `invitedclubs.com`)
회원이다. 현재는 각 골프장 사이트에 **사람마다·골프장마다 다른 아이디로 직접 로그인**(2명 × 3클럽 = 6세트)해
티타임을 일일이 조회·예약해야 한다. 번거롭고, 함께 칠 동반자 정보도 매번 새로 입력해야 한다.

**원하는 모습**: 카카오톡에서 비서처럼 대화하며 빈 티타임을 조회하고, 자주 함께 치는 동반자를 불러
그 자리에서 예약까지 끝내는 것.

---

## 2. 목표 & 성공 지표

**목표**
- G1. 카카오톡 대화만으로 3개 클럽의 빈 티타임을 **조회**한다.
- G2. 저장된 동반자(멤버/게스트)를 지정해 **예약을 실제로 완료**한다(반자동: 사용자 확인 후 앱이 실행).
- G3. 자주 함께 치는 사람을 **동반자 DB**에 저장해 매번 재입력하지 않는다.
- G4. 부부 2명이 각자의 카카오톡으로, 각자의 invited 계정으로 사용한다.

**성공 지표** — *"턴(turn)" 정의: 사용자가 보낸 메시지 1건 = 1턴. 봇의 응답·버튼 카루셀은 턴에 포함하지 않는다.*
- S1. **조회**: 첫 조회 메시지 → 빈 슬롯 표시까지 **사용자 3턴 이내**.
- S2. **예약**: 슬롯 선택 메시지 → [예약] 탭까지 **사용자 3턴 이내** (S1과 구간이 겹치지 않음).
- S3. **성능 목표**: 세션 재사용 시 결과를 **약 30초 이내** 수신 (카카오 콜백 하드리밋과 별개 — 10절·FR-9 참조).
- S4. 동반자 1명 재사용 시 이름 재입력 **0회**.

## 3. 비목표 (v1에서 하지 않음)

- N1. 조건 무인 자동예약("스나이핑")·티타임 열림 푸시 알림 → 차기

- **N5. 예약 취소·변경 → 차기.** v1에서 취소·변경은 invited 사이트에서 수동으로 한다. (FR-8 안내 메시지에 명시)

## 5. 사용자 시나리오 (대표 스토리)

- **US-1 (조회)**: "이번 토요일 코토 아침 티타임 있어?" → 봇이 빈 슬롯을 버튼으로 보여준다.

## 6. 기능 요구사항 (Functional Requirements)

각 요구사항은 ID·설명·수용 기준(AC)으로 기술한다.

### FR-1 접근 제어 (화이트리스트)
- 화이트리스트에 등록된 카카오 사용자 ID(2개)만 봇을 사용할 수 있다.
- **AC1**: 미등록 사용자가 말을 걸면 기능이 동작하지 않고 거절 메시지를 받는다.
- **AC2**: 화이트리스트 사용자가 **자격증명이 없는 클럽**을 요청하면 "이 클럽 자격증명 미등록" 안내를 받고 자동화는 시도되지 않는다.

### FR-2 자격증명 관리 & 온보딩
- 사용자별·클럽별 invited 아이디/비밀번호를 등록·갱신할 수 있고, **암호화 저장**된다.
- 최초 설정: 각자의 `kakao_user_id`를 확인해 화이트리스트에 등록하고, 6세트 자격증명을 **시드 스크립트 또는 보호된 관리 엔드포인트(키 보유 측=워커)** 로 등록한다.
- "도움말/help" 발화 시 봇이 할 수 있는 일을 안내한다.
- **AC1**: 평문 비밀번호가 DB·로그·**실패 HTML 스냅샷**에 남지 않는다(스냅샷 저장 전 비밀번호 필드 마스킹). 검증: 알려진 테스트 비밀번호로 DB 컬럼·모든 로그/스냅샷 산출물을 grep 했을 때 **0건**.
- **AC2**: 비번 변경 시 갱신할 수 있고, 이후 로그인에 즉시 반영된다.

### FR-3 자연어 의도 파싱 (NLU)
- 자유 발화에서 `의도(조회/예약/동반자관리/도움말)`, `클럽`, `날짜`, `시간대`, `인원`, `홀수`, `이동수단`을 추출한다.
- **시간대·날짜는 클럽 현지 시간(America/Los_Angeles) 기준**으로 해석하고, NLU는 **절대 날짜(YYYY-MM-DD, 클럽 현지)** 를 출력한다. (사용자 단말이 KST여도 영향받지 않음 — 시차 ~16–17h로 "이번 토요일"이 한 주 어긋나는 것 방지)
- 해석 규칙(테스트 가능): `오전`=12:00 이전, `오후`=12:00 이후(`아침`≈오전), `이번 ○요일`=현지 기준 **미래의 가장 가까운 해당 요일**(오늘이 그 요일이고 시간이 안 지났으면 오늘 포함), `홀수` 미지정 시 기본 18홀, `이동수단` 미지정 시 클럽 기본값.
- `의도/클럽/날짜` 중 하나라도 특정 불가하면 **모호**로 보고 되묻는다.
- **AC (예시, 기준 오늘=2026-06-24 수)**:
  | 발화 | intent | club | date | time | players | holes |
  |---|---|---|---|---|---|---|
  | "토욜 아침 코토 와이프랑" | lookup | Coto de Caza | 2026-06-27 | 오전 | 2(본인+아내) | 18 |
  | "내일 오후 알리소" | lookup | Aliso Viejo | 2026-06-25 | 오후 | 미지정 | 18 |
  | "다음주 일요일 올드랜치 4명 9홀" | lookup | Old Ranch | 2026-07-05 | 미지정→되묻기 | 4 | 9 |
  | "주말에 라운드 잡아줘" | (모호) | — | — | — | — | — → "어느 클럽, 토/일 중 언제로 할까요?" |

### FR-4 빈 티타임 조회
- 해당 사용자·클럽 계정으로 invited에 로그인해, 조건(날짜/시간대)에 맞는 빈 슬롯을 조회해 버튼으로 표시한다.
- **AC1 (사전 검증, 워커 호출 전 스킬서버에서)**: ① 과거 날짜/이미 지난 시간 → 거절 후 가장 가까운 유효 날짜 제안. ② `booking_window_days` 초과(아직 안 열림) → "○월 ○일부터 예약 가능" 안내.
- **AC2 (빈자리 없음)**: 0건이면 같은 날 가장 가까운 가용 슬롯 최대 3개를 제시하고, 그날 없으면 가용한 다음 날짜를 안내한다.

### FR-5 동반자 DB
- 동반자를 추가/조회/수정/삭제한다. 필드: `이름`, `멤버/게스트 구분`(필수), `멤버번호`·`태그`·`메모`(선택).
- 동반자 정보는 부부 **공용**으로 조회된다(의도된 공유 — 9절 개인정보 주석 참조).
- **AC**: 저장된 동반자가 이후 예약 시 버튼으로 노출된다.

### FR-6 동반자 선택 / 즉석 게스트 / 필수 필드
- 예약 시 저장 동반자를 버튼으로 선택하거나, 즉석에서 게스트를 추가한다.
- **AC1**: 멤버/게스트 구분이 예약에 반영된다.
- **AC2**: 예약 폼이 요구하는 필드(예: 멤버 동반자의 멤버번호, 게스트 이름)가 동반자 레코드에 없으면 **되물어 입력받는다**(조용한 실패 금지). 폼이 요구하는 정확한 필드는 M0에서 확인.

### FR-7 예약 실행 (반자동)
- 슬롯·동반자·홀수·이동수단이 확정되고 사용자가 **확인([예약])** 하면, 앱이 invited에 실제 예약을 수행한다.
- **AC1**: 사용자 확인 없이는 예약이 발생하지 않는다.
- **AC2 (인원 검증)**: 총 인원(예약자 본인 + 선택 동반자)과 NLU `player_count`를 대조하고, **포섬 최대(본인 포함 4인)** 및 클럽별 그룹/게스트 제한을 초과하면 차단·안내하며 자동화를 실행하지 않는다. (클럽 제한값은 어댑터 파라미터로 M0 확인)
- **AC3 (신선도)**: `conversations.state`에 TTL(예: 5–10분)을 둔다. 만료/오래된 선택에서 [예약]을 누르면 자동 진행하지 않고 재조회한다. 실행 **직전 슬롯 가용성을 재확인**한다.
- **AC4 (배우자 충돌)**: 확정 직전 `(club_id, tee_datetime)` 예약을 조회해 배우자가 같은 슬롯을 `requested/confirmed` 상태로 갖고 있으면 "배우자가 이미 이 슬롯을 예약(시도) 중"이라 경고하고 재확인을 받는다.
- **AC5 (중복 방지/멱등)**: `bookings`에 `user_id+club_id+tee_datetime` 부분 유니크 키. 결과 불명/타임아웃 시 **자동 재시도 금지**, "예약 상태 확인 필요"로 표시하고, 재시도 전 동일 슬롯 예약 존재 여부를 먼저 확인한다.

### FR-8 결과 보고
- **성공 판정은 확인번호와 독립**: 제출 후 검증(확인 페이지 감지 **또는** 계정 예약 목록에 노출)으로 성공을 판정한다. 확인번호가 있으면 포함, 없으면 "확인번호 미제공"이라고 명시.
- 실패 시 **분류된 사유**를 안내한다: `자격증명 오류` / `슬롯 마감` / `봇 차단·캡차` / `사이트 구조 변경` / `타임아웃` / `기타`.
- 취소·변경은 v1 비목표(N5)임을 안내: "취소는 invited 사이트에서 직접 해주세요."
- **AC**: 성공/실패가 명확히 구분되어 전달되고, 실패는 위 분류 중 하나로 안내된다.

### FR-9 비동기 콜백 처리 (결정적 규칙)
- 스킬서버는 인바운드 웹훅 수신 후 **항상 N초(예: 4초) 이내 "처리 중…" useCallback 응답**을 반환하고(작업 소요시간을 미리 예측하지 않음), 결과는 콜백으로 전달한다.
- **카카오 콜백 하드리밋 ≈ 1분.** 1분 초과 시 결과를 저장하고, 사용자가 "결과"로 재조회한다.
- **AC**: 40초 걸리는 조회에서 사용자는 즉시 "처리 중"을, 60초 이내 결과 콜백을 받는다. 60초 초과 시 "결과" 재조회로 저장된 결과를 받는다.

### FR-10 다계정 라우팅
- 말하는 사람(카카오ID)에 따라 해당 사용자의 invited 계정·자격증명을 사용한다.
- **AC**: 당신/아내가 각자 자기 계정으로 동작한다. (자격증명 없는 클럽은 FR-1 AC2 적용)

### FR-11 로그인 세션 재사용 (성능 최적화)
- 로그인 쿠키/세션(`storage_state`)을 암호화 저장·재사용해 재로그인 빈도를 줄인다.
- **유효 세션 정의(운영적)**: `expires_at`이 미래 **그리고** 경량 인증 프로브가 "로그인됨"을 반환.
- **AC**: `job_logs.auth_mode`에 `session_reuse`(프로브 통과) 또는 `full_login`을 기록해 관측 가능. (세션 재사용 가능 여부·수명은 M0에서 확인 — R7)

### FR-12 예약 이력 (도구 발생분 한정)
- 도구를 통해 발생한 예약 요청·완료·실패를 로그로 저장한다(감사·멱등에 필요, M2).
- invited 사이트에서 직접 한 예약은 범위 밖.
- 대화형 "내 예약" 조회 UI는 **선택(2차)** — 핵심 조회/예약 경로 다음 우선순위(M3 이후).

## 7. 비기능 요구사항 (Non-Functional)

- **NFR-보안**: 자격증명·세션 쿠키는 AES-256-GCM 암호화. 복호화 키 `ENC_KEY`는 워커 환경변수에만 보관(DB·Vercel에 없음). 스킬서버↔워커는 공유 시크릿으로 상호 인증. 전 구간 HTTPS.
  - **키 관리 주석**: `ENC_KEY`는 안전한 곳에 백업한다. 분실 시 저장된 자격증명·세션을 복호화할 수 없어 **6세트를 재시드**해야 한다.
- **NFR-개인정보**: 2인 전용. 동반자·자격증명은 최소한으로 보관(멤버번호 포함). 동반자 레코드는 의도적으로 부부 양쪽에 공개.
- **NFR-신뢰성**: 자동화 실패 시 **안전 실패**(임의 예약 금지). **일시 오류만** 백오프 재시도. **CAPTCHA/OTP/기기 인증을 만나면 재시도하지 않고 즉시 중단·사용자 통지**(재시도는 봇차단을 악화). 실패 시 (마스킹된) HTML 스냅샷 로깅.
- **NFR-성능**: 측정 구간 = 인바운드 웹훅 수신 → 콜백 전달. 목표: **세션 재사용 < 60초(체감 ~30초)**, **콜드 로그인(전체 Playwright 로그인+봇차단 통과) < 120초**. 초과 시 저장-후-"결과" 흐름으로 폴백.
- **NFR-동시성**: 같은 계정 작업은 **자격증명 단위로 직렬화(짧은 큐/락)**, 사용자별 독립 `BrowserContext`, `sessions` 갱신은 조건부(updated_at/version)로 last-write-wins 손상 방지. (2인 저빈도라 "자격증명별 순차 처리 + 짧은 큐"로 충분)
- **NFR-비용**: 2인용으로 무료~소액. NLU는 저비용 `claude-haiku-4-5`.
- **NFR-유지보수**: 클럽별 셀렉터·플로우를 **어댑터 모듈**로 격리(사이트 변경 시 한 곳만 수정).
- **NFR-준법/ToS**: invited 약관상 자동화 제한 가능성 고지. 저빈도 개인 사용 전제, 차단 시 즉시 중단·수동 전환.

## 8. 시스템 구성 / 핵심 기술 결정

```
            ┌───────────────────────────── ⑥ 콜백 ──────────────────────────────┐
            ▼                                                                     │
[카카오톡] ─webhook→ [Vercel: 스킬서버(Next.js/TS)] ─HTTP→ [워커(Render, Node/Playwright)] ─→ [Supabase]
 (당신/아내)          ① 화이트리스트  ② NLU(Claude)         ④ invited 로그인(세션재사용/프로브)
                     ③ useCallback "처리중"                  ⑤ 조회/예약 실행
```

- **인터페이스**: 카카오톡 챗봇(대화+버튼 하이브리드). 복잡한 선택은 버튼/카루셀.
- **자동화**: invited 공개 API 없음 → Playwright 헤드리스 자동화. 로그인은 아이디/비번만(OTP·캡차 없음 전제).
- **전제 깨짐 폴백**: CAPTCHA/OTP/기기 인증 발생 시 → 재시도 없이 중단·통지. 최소 폴백안: 사용자가 카카오로 OTP를 전달해 워커가 입력 / 또는 사람이 사전 로그인된 `storage_state`를 주기적으로 주입.
- **실행 환경**: Vercel(스킬서버·DB연동) + 전용 워커(Render 기본, Playwright) + Supabase(Postgres).
- **NLU**: Claude API `claude-haiku-4-5`.
- **비동기**: 카카오 콜백(약 1분). 알림톡/사업자등록 불필요.

## 9. 데이터 요구사항 (요약)

- **users** (kakao_user_id, email, display_name)
- **clubs** (name, **provider** — *v1=invited만, trinity는 차기 확장값(N2)*, timezone, booking_window_days, 그룹/게스트 제한 등 어댑터 파라미터)
- **credentials** (user×club, 암호화된 username/password, status) — unique(user, club)
- **sessions** (credential별 암호화 storage_state, expires_at, updated_at/version)
- **companions** (name, type[member/guest], member_number?, tags?, note?) — 부부 공용(의도된 공유, 최소 보관)
- **bookings** (user, club, tee_datetime, players[], **holes**, **transport**, status, confirmation_ref?, error?) — `status='cancelled'`은 차기용 예약값(v1 미사용), `user_id+club_id+tee_datetime` 부분 유니크
- **conversations** (user, state, **ttl/expires_at**) — 멀티턴 진행 상태
- **job_logs** (type[lookup/book], params, status, result, error, **auth_mode**) — 감사/재시도

상세 스키마는 설계 문서 6절 참조.

## 10. 외부 의존성 & 제약

- **invited**: 공개 API 없음(자동화 의존), ToS상 자동화 제한 가능, 사이트 구조 변경 위험.
- **카카오 챗봇**: 동기 응답 ~5초, **콜백 하드리밋 ~1분**. 채널+오픈빌더 필요, 화이트리스트로 사적 사용.
- **Vercel**: 서버리스 실행시간·헤드리스 브라우저 한계 → 자동화는 워커로 분리.
- **Supabase / Render / Claude API**: 각 서비스 가용성·요금에 의존.

## 12. 리스크 & 미해결 질문 (스파이크에서 해소)

- **R1**: invited 봇 탐지(Cloudflare 등) → 차단 시 자동화 불가. M0 확인.
- **R2**: 사이트 구조 변경으로 자동화 깨짐 → 어댑터로 영향 최소화.
- **R3**: 클럽별 예약 오픈 N일·조회 가능 범위 미확정 → M0 확인.
- **R4**: 예약 폼의 동반자(멤버/게스트)·홀수·카트·멤버번호 입력 필드·검색 방식 미확정 → M0 확인.
- **R5**: 예약 확인번호 취득 가능 여부 미확정 → M0 확인(미취득이어도 FR-8 검증으로 성공 판정).
- **R6**: 콜백 1분 초과 작업의 "결과" 재조회 UX가 충분한지 → M1에서 실제 조회 소요시간 측정.
- **R7**: 로그인 세션 재사용 가능 여부·유효기간 미확정 → M0 확인(불가 시 콜드로그인 예산으로 운영).
