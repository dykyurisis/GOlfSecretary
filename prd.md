## 1. 배경 / 문제 정의

부부 2명이 미국 오렌지카운티 회원제 골프장 3곳(**Coto de Caza, Aliso Viejo, Old Ranch**, 모두 `invitedclubs.com`)
회원이다. 현재는 사람마다·골프장마다 다른 아이디로 각 사이트에 직접 로그인(2명 × 3클럽 = 6세트)해
티타임을 일일이 조회·예약하고, 동반자 정보도 매번 새로 입력한다.

**원하는 모습(1단계)**: **아이폰에서 쓰기 좋은 웹 대시보드**로 한 곳에서 빈 티타임을 조회하고,
자주 함께 치는 동반자를 불러 예약까지 끝낸다. **카카오톡 대화형은 2단계**에서 같은 백엔드로 추가한다.

---

## 2. 목표 & 성공 지표

**목표**
- G1. 웹 대시보드에서 3개 클럽의 빈 티타임을 **조회**(폼/버튼 + 챗 보조).
- G2. 저장된 동반자(멤버/게스트)를 지정해 **예약을 실제로 완료**(반자동: 사용자 확인 후 실행).
- G3. 자주 함께 치는 사람을 **동반자 DB**에 저장해 재입력하지 않는다.
- G4. 부부 2명이 **각자 Google 로그인**으로, 각자의 invited 계정으로 사용한다.
- G5. **아이폰 모바일 우선 PWA**(홈화면 추가).

**성공 지표**
- S1. 조회 폼 제출/챗 입력 → 빈 슬롯 표시까지 **조작 3스텝 이내**.
- S2. 슬롯 선택 → [예약] 확정까지 **조작 3스텝 이내**.
- S3. 결과 수신 목표: 세션 재사용 시 **약 30초 이내**, 콜드 로그인 < 120초(진행 표시 + 실시간 갱신).
- S4. 동반자 1명 재사용 시 이름 재입력 **0회**.

## 6. 기능 요구사항 (Functional Requirements)

### FR-1 인증 & 권한 (Google + 허용목록)
- Supabase Auth Google OAuth로 로그인. **허용 Gmail 2개**만 접근, 그 외 거부.
- RLS로 데이터 접근을 통제(자신의 credentials/bookings/jobs, 공용 companions/clubs).
- **AC1**: 허용목록 외 Google 계정은 로그인 후에도 데이터·기능에 접근 불가.
- **AC2**: 자격증명이 없는 클럽을 요청하면 "이 클럽 자격증명 미등록" 안내, 자동화 미실행.

### FR-2 자격증명 관리 & 온보딩
- 사용자별·클럽별 invited 아이디/비밀번호를 설정 화면에서 등록·갱신(암호화 저장, 키는 워커 보유 측 경유).
- 최초 설정: 허용 Gmail 등록(상수/환경설정), 6세트 자격증명 시드. "도움말"로 사용법 안내.
- **AC1**: 평문 비밀번호가 DB·로그·**실패 HTML 스냅샷**에 남지 않는다(스냅샷 전 마스킹). 검증: 테스트 비밀번호로 DB 컬럼·로그/스냅샷 grep 시 **0건**.
- **AC2**: 비번 변경 시 갱신되어 이후 로그인에 즉시 반영.

### FR-3 자연어 챗 보조 (NLU)
- 챗 입력의 자유 발화에서 `의도(조회/예약/동반자/도움말)`, `클럽`, `날짜`, `시간대`, `인원`, `홀수`, `이동수단`을 추출해 **폼을 자동으로 채운다**.
- **시간대·날짜는 클럽 현지(America/Los_Angeles) 기준**, NLU는 **절대 날짜(YYYY-MM-DD, 현지)** 출력. (KST 단말이어도 영향 없음)
- 규칙(검증 가능): `오전`=12:00 이전, `오후`=12:00 이후(`아침`≈오전), `이번 ○요일`=현지 미래 가장 가까운 해당 요일, `홀수` 미지정 시 18, 모호 항목은 폼에서 비워 표시.
- **AC**: 챗으로 채워진 폼을 **사용자가 화면에서 확인·수정한 뒤** 실행한다(자동 실행 금지). 예시:
  | 발화(오늘=2026-06-24 수) | club | date | time | players | holes |
  |---|---|---|---|---|---|
  | "토욜 아침 코토 와이프랑" | Coto de Caza | 2026-06-27 | 오전 | 2 | 18 |
  | "내일 오후 알리소" | Aliso Viejo | 2026-06-25 | 오후 | 미지정 | 18 |
  | "다음주 일요일 올드랜치 4명 9홀" | Old Ranch | 2026-07-05 | 미지정(빈칸) | 4 | 9 |

### FR-4 빈 티타임 조회
- 해당 사용자·클럽 계정으로 invited에 로그인해 조건(날짜/시간대)에 맞는 빈 슬롯을 조회·표시(현지 시간 표기).
- **AC1 (사전 검증, 워커 호출 전)**: ① 과거 날짜/지난 시간 → 거절+가장 가까운 유효 날짜 제안. ② `booking_window_days` 초과 → "○월 ○일부터 예약 가능".
- **AC2 (빈자리 없음)**: 0건이면 같은 날 가장 가까운 가용 슬롯 최대 3개, 없으면 가용한 다음 날짜 안내.

### FR-5 동반자 DB
- 동반자 추가/조회/수정/삭제. 필드: `이름`, `멤버/게스트`(필수), `멤버번호`·`태그`·`메모`(선택). **부부 공용**(RLS).
- **AC**: 저장된 동반자가 예약 화면에서 칩/버튼으로 노출.

### FR-6 동반자 선택 / 즉석 게스트 / 필수 필드
- 예약 시 저장 동반자를 선택하거나 즉석 게스트를 추가.
- **AC1**: 멤버/게스트 구분이 예약에 반영.
- **AC2**: 예약 폼이 요구하는 필드(멤버번호·게스트 이름 등)가 동반자 레코드에 없으면 **되물어 입력**(조용한 실패 금지). 정확한 필드는 M0 확인.

### FR-7 예약 실행 (반자동)
- 슬롯·동반자·홀수·이동수단 확정 후 사용자가 **[예약]** 을 누르면 앱이 invited에 실제 예약.
- **AC1**: 사용자 확인 없이는 예약이 발생하지 않는다.
- **AC2 (인원 검증)**: 총 인원(본인+동반자)과 `player_count` 대조, **포섬 최대(본인 포함 4인)** 및 클럽 제한 초과 시 차단·안내, 자동화 미실행.
- **AC3 (신선도)**: `chat_sessions`/선택 상태에 TTL(5–10분). 만료·오래된 선택으로 실행 시 재조회. 실행 **직전 슬롯 가용성 재확인**.
- **AC4 (배우자 충돌)**: 확정 직전 `(club_id, tee_datetime)` 예약 조회 → 배우자가 같은 슬롯을 `requested/confirmed`면 경고·재확인.
- **AC5 (멱등/중복 방지)**: `bookings`에 `user_id+club_id+tee_datetime` 부분 유니크. 결과 불명/타임아웃 시 **자동 재시도 금지**, "예약 상태 확인 필요" 표시, 재시도 전 동일 슬롯 존재 확인.

### FR-8 결과 보고
- **성공 판정은 확인번호와 독립**: 제출 후 검증(확인 페이지 감지 **또는** 계정 예약 목록 노출). 확인번호 있으면 포함, 없으면 "확인번호 미제공" 명시.
- 실패 시 **분류 사유** 안내: `자격증명 오류`/`슬롯 마감`/`봇 차단·캡차`/`사이트 구조 변경`/`타임아웃`/`기타`.
- 취소·변경은 비목표(N4): "취소는 invited 사이트에서 직접" 안내.
- **AC**: 성공/실패가 명확히 구분, 실패는 위 분류 중 하나.

### FR-9 비동기 잡 & 실시간 갱신
- 조회/예약은 `jobs`에 `queued`로 생성 → 워커가 처리(`running`→`done/failed`) → 대시보드가 **Supabase Realtime/폴링**으로 진행·결과 갱신.
- 카카오 콜백 1분 한계 없음. 장시간 작업도 진행 표시로 자연스럽게 대기.
- **AC**: 사용자는 "조회/예약 중…"을 즉시 보고, 완료 시 결과가 자동 갱신된다(새로고침 불필요).

### FR-10 다계정 라우팅
- 로그인한 사용자에 따라 해당 사용자의 invited 계정·자격증명을 사용.
- **AC**: 당신/아내가 각자 자기 계정으로 동작(없는 클럽은 FR-1 AC2).

### FR-11 로그인 세션 재사용 (성능 최적화)
- `storage_state`를 암호화 저장·재사용해 재로그인 빈도 감소.
- **유효 세션 정의**: `expires_at` 미래 **그리고** 경량 인증 프로브가 "로그인됨" 반환.
- **AC**: `jobs.auth_mode`에 `session_reuse`/`full_login` 기록(관측 가능). 가능 여부·수명은 M0 확인(R7).

### FR-12 예약 이력 (도구 발생분 한정)
- 도구로 발생한 예약 요청·완료·실패를 로그로 저장(감사·멱등). invited에서 직접 한 예약은 범위 밖.
- 이력 화면(설정/이력)에서 최근 예약 조회.

### FR-13 PWA / 모바일
- 아이폰 모바일 우선 반응형. 매니페스트로 **홈화면 추가(PWA)** 가능. (오프라인 작동은 v1 비목표 — 네트워크 필요)
- **AC**: 아이폰 Safari에서 홈화면 추가 후 앱처럼 실행되고, 주요 흐름이 모바일 화면에 맞게 보인다.

## 7. 비기능 요구사항 (Non-Functional)

- **NFR-인증/권한**: Supabase Auth Google + 허용 Gmail 2개. RLS로 행 단위 접근 통제.
- **NFR-보안**: 자격증명·세션 AES-256-GCM 암호화, 키 `ENC_KEY`는 워커 환경변수에만(DB·Vercel에 없음). 대시보드↔워커 공유 시크릿. 전 구간 HTTPS.
  - **키 관리**: `ENC_KEY`는 안전히 백업. 분실 시 6세트 재시드 필요.
- **NFR-개인정보**: 2인 전용. 동반자·자격증명 최소 보관(멤버번호 포함). 동반자는 의도적으로 부부 공용.
- **NFR-신뢰성**: 자동화 실패 시 **안전 실패**(임의 예약 금지). 일시 오류만 백오프 재시도. **CAPTCHA/OTP/기기 인증 시 재시도 없이 즉시 중단·통지**. 실패 시 마스킹된 HTML 스냅샷 로깅.
- **NFR-성능**: 측정 구간=요청→결과 갱신. 세션 재사용 < 60초(체감 ~30초), 콜드 로그인 < 120초. 진행 상태 실시간 표시.
- **NFR-동시성**: 같은 계정 작업은 **자격증명 단위 직렬화**(짧은 큐/락), 사용자별 독립 BrowserContext, `sessions` 갱신은 조건부(updated_at/version).
- **NFR-비용**: 2인용 무료~소액. NLU는 `claude-haiku-4-5`.
- **NFR-유지보수**: 클럽별 셀렉터·플로우를 **어댑터 모듈**로 격리.
- **NFR-준법/ToS**: invited 약관상 자동화 제한 가능. 저빈도 개인 사용, 차단 시 중단·수동 전환.

## 8. 시스템 구성 / 핵심 기술 결정

```
[아이폰 브라우저/PWA] ⇅ Supabase Realtime/폴링(잡 상태)
        │
        ▼
[Vercel: Next.js(TS)]  ── 잡 생성(queued) ──▶ [Supabase(Postgres+Auth+Realtime)]
  · Google 로그인+허용목록                          ▲   잡 갱신/결과
  · 폼 API/서버액션, 챗 NLU(Claude)→폼 프리필         │
        │  잡 픽업/HTTP 트리거(공유 시크릿)            │
        ▼                                          │
[워커: Render(Node/Playwright)] ── 로그인·조회·예약 ──┘
```

- **인터페이스**: 웹 대시보드(폼/버튼 + 챗 보조), 아이폰 모바일 우선 PWA.
- **인증**: Supabase Auth Google, 허용 Gmail 2개.
- **자동화**: invited 공개 API 없음 → Playwright(아이디/비번 로그인). 워커(Render)에서 실행.
- **비동기**: `jobs` 큐 + Supabase Realtime(카카오 콜백 제약 없음).
- **NLU**: Claude `claude-haiku-4-5` (폼 프리필, 사용자 확인 후 실행).

## 9. 데이터 요구사항 (요약)

- **users** (id=auth uid, email, display_name) — 허용 이메일은 상수/환경설정
- **clubs** (name, provider[invited|trinity(차기)], timezone, booking_window_days, 그룹/게스트 제한)
- **credentials** (user×club, 암호화 username/password, status) — unique(user, club)
- **sessions** (credential별 암호화 storage_state, expires_at, updated_at/version)
- **companions** (name, type[member/guest], member_number?, tags?, note?) — 부부 공용(RLS)
- **bookings** (user, club, tee_datetime, players[], holes, transport, status, confirmation_ref?, error?) — `cancelled`은 차기용, `user+club+tee_datetime` 부분 유니크
- **jobs** (type[lookup/book], params, status[queued/running/done/failed], result, error, auth_mode) — 큐+감사, Realtime 구독
- **chat_sessions** (user, state, expires_at[TTL]) — 챗 보조 맥락

상세 스키마·RLS는 설계 문서 6절 참조.

## 10. 외부 의존성 & 제약

- **invited**: 공개 API 없음(자동화 의존), ToS상 자동화 제한 가능, 구조 변경 위험.
- **Supabase**: Auth(Google)·DB·Realtime 의존.
- **Vercel**: 서버리스 실행시간·헤드리스 브라우저 한계 → 자동화는 워커로 분리.
- **Render / Claude API**: 가용성·요금 의존.
- **아이폰 PWA**: iOS Safari의 PWA 제약(푸시·백그라운드 제한) — v1은 포그라운드 사용 전제.

## 12. 리스크 & 미해결 (M0에서 해소)

- **R1**: invited 봇 탐지(Cloudflare 등) → 차단 시 자동화 불가.
